# AI Coding Rules — AI Guitar Tutor

These rules apply to every AI assistant working in this repository.
Read `.architecture.md` and `artifacts.md` first — they describe what is actually built.

---

## Project Identity

AI Guitar Tutor is a **production web app** deployed on Vercel + Render.
It is not a prototype. Every change must keep the live app working.

- Frontend: Next.js 14 App Router, TypeScript, Tailwind, shadcn/ui, Framer Motion, SWR
- Backend: FastAPI, Python 3.12, SQLAlchemy 2.0 async, Groq LLM, librosa
- Database: PostgreSQL (production) / SQLite (local dev via aiosqlite)
- Auth: JWT access + refresh tokens, roles: `student` | `teacher` | `admin`

---

## Core Rules

### 1. Never Break Existing Functionality
- Before modifying any file, read it in full.
- Understand how the file is used by other components before changing its interface.
- Do not rename exported functions, component props, or API endpoint paths without updating all callers.
- Do not change Pydantic model field names without updating the frontend TypeScript interfaces in `frontend/src/lib/api.ts`.

### 2. TypeScript Is Strict
- All new frontend code must be TypeScript. No `any` unless there is a compelling reason.
- API response types are defined in `frontend/src/lib/api.ts`. Add new interfaces there — do not inline them.
- Framer Motion `ease` values must use the `Easing` type from `framer-motion`. Do not use bare strings like `"easeOut"` — use the variants already in `lib/motion.ts`.

### 3. Python Type Hints Are Required
- All Python functions must have full type hints.
- Use `from __future__ import annotations` at the top of every new Python file.
- Pydantic models must use `model_copy(update={...})` not direct mutation.

### 4. Database Changes Need Migrations
- Never modify `db/models.py` without creating an Alembic migration.
- Generate with: `cd backend && alembic revision --autogenerate -m "description"`
- Test the migration locally before committing.

### 5. Async All the Way Down (Backend)
- All database operations must use `async with` / `await`. Never call `.execute()` synchronously.
- Use `AsyncSession` from SQLAlchemy. Never use a sync session.
- FastAPI endpoints that touch the DB must be `async def`.

### 6. No Global State on the Frontend
- Do not introduce Redux, Zustand, or any global store. Use React Context only for auth.
- Component-level state: `useState`. Remote data: SWR.
- SWR cache keys must match the API path exactly (e.g. `"/api/progress"`).

### 7. API Design
- New backend endpoints go in their own router file, not in `main.py`.
- Register new routers in `main.py` with `app.include_router(...)`.
- All responses must be JSON. Always use Pydantic response models.
- Auth-protected endpoints must use `Depends(get_current_user)` or `Depends(require_role("role"))`.
- Do not hard-code user IDs. Always extract from the JWT via the dependency.

### 8. Groq / LLM Usage
- The Groq client is initialised in `feedback_engine/generator.py` via `_get_groq_client()`. Reuse this — do not create new clients.
- Use the model constant `GROQ_MODEL` (currently `llama-3.3-70b-versatile`). Do not hardcode model strings.
- All LLM calls must have a fallback for when `GROQ_API_KEY` is not set. Never let a missing key crash the app.
- Keep prompts focused and under 300 words. Specific prompts produce better outputs than vague ones.

### 9. Audio Processing
- All audio evaluation lives in `backend/audio_engine/`. Do not add signal processing logic to `main.py` or any router.
- Audio files are always temporary — written to `tempfile.NamedTemporaryFile` and deleted after evaluation.
- Never assume the audio format. Use `librosa.load()` which handles wav/webm/ogg automatically.

### 10. Component Architecture (Frontend)
- New pages go in `frontend/src/app/` using the Next.js App Router convention.
- Shared UI primitives go in `frontend/src/components/ui/`. Use shadcn/ui components where possible.
- Feature-specific components go in their feature folder (e.g. `SongCouncil/`, `feedback/`, `AdminPanel/`).
- All interactive components need `"use client"` at the top. Static/layout components do not.
- All data-driven surfaces must have three states: loading (`<LoadingSkeleton>`), error (`<ErrorState>`), and data.
- Always use `AppShell` as the layout wrapper for authenticated pages.

### 11. Design System
- Use the existing glassmorphism design system. Key classes: `glass-card`, `btn-gradient`, `gradient-text`.
- Use the colour palette already in `tailwind.config.ts`. Do not introduce new one-off hex values without adding them to the config.
- Use Framer Motion variants from `lib/motion.ts` (`fadeInUp`, `staggerContainer`, `cardEntrance`). Do not write inline animation objects.
- All animations must call `useReducedMotion()` and disable/simplify when it returns `true`.

### 12. CORS & Security
- The `ALLOWED_ORIGINS` environment variable controls CORS in production. Never hardcode frontend URLs in the backend.
- Never log JWT tokens, passwords, or full request bodies.
- Never expose the `password_hash` field from the `User` model in any API response.
- Refresh tokens are stored hashed in the DB — never return the raw token after the initial issue.

## File-Specific Rules

### `backend/main.py`
- Only add endpoints here if they are truly core (session management, health check, audio submit).
- Everything else belongs in a dedicated router.
- The `_fingerings_by_chord` dict is loaded once at startup — do not reload it per-request.

### `backend/council/schemas.py`
- `LessonDocument` is the contract between backend and frontend. Field name changes are breaking.
- Add new optional fields with `Optional[X] = None` to avoid breaking existing lesson sessions.

### `backend/db/models.py`
- Add new columns as nullable or with a default to avoid migration failures on existing data.
- All UUID primary keys use `PG_UUID(as_uuid=True)` for Postgres compatibility.

### `frontend/src/lib/api.ts`
- This is the single source of truth for all TypeScript API types.
- Every new backend endpoint needs a corresponding function and interface here.
- The `request<T>()` helper handles auth headers, token refresh, and error throwing — always use it.

### `frontend/src/components/SongCouncil/IntegratedLessonView.tsx`
- This component owns the `activeSection` / `highlightedChords` state for the entire lesson view.
- Section click → `handleScrollToSection()` → updates state + scrolls to `chordBoardRef`.
- Do not lift this state further up without strong reason.

## What To Avoid

- Do not use `localStorage` for new features — use the DB-backed progress API.
- Do not use `console.log` in production code — use `logging.getLogger(__name__)` (backend) or remove logs.
- Do not use inline styles for colours that belong in the design system.
- Do not create new LLM agent files without following the pattern in `council/agents.py`.
- Do not add npm packages without checking if a shadcn/ui component or existing dependency already covers the need.
- Do not use `git push --force` on `main` — it deploys immediately to production.
- Do not store secrets in code. All secrets live in environment variables.

## Testing

### Backend
- Tests live in `backend/tests/`. Use `pytest` with `conftest.py` fixtures.
- New endpoints should have at least a smoke test in `tests/test_api.py`.
- Run: `cd backend && pytest tests/ -v`

### Frontend
- Tests live in `frontend/src/lib/__tests__/`. Use `vitest`.
- New utility functions in `lib/` should have corresponding tests.
- Run: `cd frontend && npx vitest run`

---

## Deploying

Push to `main`. That's it.
```bash
git push origin main
```
Vercel rebuilds frontend. Render rebuilds backend. Check both dashboards for build errors.
