# AI Coding Rules for AI Guitar Tutor

This file defines guidelines for AI code generation inside this repository.
When generating code, modifying files, or suggesting architecture changes, follow these rules carefully.

The goal is to maintain a **clean, modular, scalable architecture** for the AI Guitar Tutor project.

---

# Project Overview

AI Guitar Tutor is an AI-powered web application that helps users learn guitar chords through:

• prerecorded lesson videos
• microphone-based chord evaluation
• automated feedback and tutoring

The system architecture separates teaching logic, audio processing, and user interaction.

The AI assistant should **respect the architecture and avoid introducing tightly coupled code**.

---

# Core System Architecture

The system is divided into five major components.

Frontend
↓
Tutor Agent (Session Controller)
↓
Lesson Service
↓
Audio Evaluation Engine
↓
Feedback Engine

Each module must remain **independent and reusable**.

---

# Architecture Principles

## 1. Modular Design

Each component must have a single responsibility.

Modules:

backend/lesson_service
Loads and manages lesson content.

backend/audio_engine
Processes recorded audio and evaluates chord accuracy.

backend/feedback_engine
Converts evaluation results into human-readable feedback.

backend/tutor_agent
Controls the tutoring session and learning loop.

frontend/
Handles UI and microphone recording.

Modules must not directly depend on each other except through clear interfaces.

---

## 2. Tutor Agent Is the Orchestrator

The Tutor Agent controls the learning flow.

It is responsible for:

• interpreting user requests
• loading lessons
• requesting audio evaluation
• generating tutor responses
• controlling session state

It should **not contain audio processing logic**.

---

## 3. Audio Processing Must Be Isolated

All audio signal processing must exist inside:

backend/audio_engine

Use Python signal-processing libraries such as:

librosa
numpy
scipy

The audio engine should output structured evaluation results.

Example output:

{
"score": 0.82,
"detected_notes": ["G","B","D"],
"missing_notes": [],
"issue": null
}

---

## 4. Feedback Engine Must Be Simple

The feedback engine converts evaluation results into human-readable guidance.

Example:

Input:

{
"issue": "muted_string",
"string": "B"
}

Output:

"Your B string isn't ringing clearly. Try pressing closer to the fret."

Keep responses:

• short
• encouraging
• beginner-friendly

---

## 5. Lesson Data Must Be File-Based

Lessons are stored in:

data/lessons/

Example structure:

data/lessons/G_major/

lesson.mp4
reference.wav
metadata.json

Example metadata:

{
"chord": "G_major",
"notes": ["G","B","D"],
"difficulty": "beginner"
}

LessonService should dynamically load lessons from disk.

# Backend Coding Standards

## Use Clean Python Architecture

Use classes for core modules.

Example classes:

LessonService
ChordEvaluator
FeedbackGenerator
TutorAgent

Avoid global state.

---

## Use Type Hints

All Python functions must use type hints.

Example:

def evaluate_chord(audio_path: str) -> dict:

---

## Logging

Use structured logging instead of print statements.

---

## Error Handling

Gracefully handle:

• missing audio
• invalid lessons
• corrupted files

Return clear error messages.

---

# Frontend Guidelines

Frontend is responsible for:

• chord request input
• video playback
• microphone recording
• sending audio to backend
• displaying feedback

Use:

React or Next.js
Web Audio API

Do not implement audio analysis in the frontend.

---

# API Design Rules

Backend APIs must be simple and predictable.

Required endpoints:

POST /learn_chord
POST /submit_audio
GET /lesson/{chord}

All responses must be JSON.

Example:

{
"feedback": "Your B string is muted."
}

---

# Session State Machine

The tutor agent must maintain session states.

States:

IDLE
TEACHING
WAITING_FOR_PLAY
ANALYZING
FEEDBACK
COMPLETED

The AI must not create uncontrolled loops or state confusion.

---

# Code Quality Rules

When generating code:

• prefer clarity over cleverness
• write readable functions
• avoid large monolithic files
• keep modules under ~300 lines if possible

Break large logic into helper functions.

---

# Testing Expectations

Whenever possible, generate simple tests.

Example:

test_audio_engine.py
test_lesson_service.py

Tests should verify:

• chord detection
• lesson loading
• feedback generation

---

# Performance Considerations

Audio evaluation should be efficient.

Avoid:

• loading large models unnecessarily
• redundant FFT calculations
• blocking frontend requests

---

# Future Scalability

The architecture must allow future expansion to:

• additional chords
• new instruments
• real-time analysis
• mobile clients

Do not hardcode guitar-only assumptions.

---

# Final Rule

The AI assistant must always respect the architecture defined in this file.

If a request conflicts with these rules, suggest a modular solution instead of breaking the architecture.
