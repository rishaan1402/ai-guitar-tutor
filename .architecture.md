# AI Guitar Tutor — Architecture

This document is the definitive technical reference for the system. It describes the actual codebase as deployed — not an aspirational design.

---

## High-Level Overview

```
Browser (Next.js 14)
        │
        │ REST + FormData (HTTPS)
        ▼
FastAPI Backend (Render)
        │
        ├── Audio Engine (librosa/scipy) — chord evaluation
        ├── Council System (Groq LLM)   — song lesson generation
        ├── Personalization Engine       — daily practice plan
        ├── Auth Layer (JWT)             — access + refresh tokens
        └── PostgreSQL (Render)          — all persistent data
```

The frontend and backend are completely decoupled — they communicate only through REST APIs. The frontend is deployed to Vercel; the backend to Render. Both auto-deploy from pushes to the `main` branch on GitHub.

### State Management
- **Auth state:** React Context (`AuthContext`) — stores user object, JWT token helpers
- **Remote data:** SWR hooks — keyed by API path, auto-revalidates
- **UI state:** `useState` local to each component — no global store
- **Session ID:** `crypto.randomUUID()` generated client-side, sent as `X-Session-Id` header on every request

### Design System
Glassmorphism aesthetic. Key CSS classes: `glass-card`, `btn-gradient`, `gradient-text`.
Design tokens live in `tailwind.config.ts`. shadcn/ui primitives are in `components/ui/`.
All animations use Framer Motion variants from `lib/motion.ts` and respect `prefers-reduced-motion`.

## Backend Architecture

### Entry Point: `main.py`
- Instantiates shared stateless services: `LessonService`, `ChordEvaluator`, `FeedbackGenerator`
- Loads `fingerings.json` into memory at startup
- Manages in-memory session store (`_sessions` dict, 1-hour TTL, cleaned on each request)
- Mounts all routers with their prefixes
- Configures CORS from `ALLOWED_ORIGINS` env var (falls back to `*` for local dev)

### Session State Machine (`tutor_agent/agent.py`)
Each browser session gets its own `TutorAgent` instance, which implements a state machine:
```
IDLE → TEACHING → WAITING_FOR_PLAY → ANALYZING → FEEDBACK → (WAITING_FOR_PLAY or COMPLETED)
```
The session ID is a UUID passed in the `X-Session-Id` header on every request.

### Audio Evaluation Pipeline (`audio_engine/`)
```
Raw audio file (wav/webm/ogg)
        ↓
librosa.load() — resample to 22050 Hz mono
        ↓
Noise reduction (scipy spectral subtraction)
        ↓
Frame segmentation + FFT
        ↓
Pitch detection (librosa.pyin / yin)
        ↓
Frequency → MIDI note → note name mapping
        ↓
Compare detected notes vs expected chord notes
        ↓
Evaluation result:
  { score, detected_notes, missing_notes, extra_notes, issue }
```

### Feedback Engine (`feedback_engine/generator.py`)
- `generate_analysis_summary(evaluation)` — human-readable summary of what happened
- `generate_fingering_tips(missing_notes, positions)` — per-note tips referencing actual finger/string data
- Uses Groq LLM for natural language; falls back to template strings if API key is absent
- Tone is adjusted based on `user.skill_level` (beginner vs advanced language)

### Council System (`council/`)
The Council is the core AI lesson generation system. It runs when a user searches for a song.

```
POST /api/council/generate
        ↓
ingestion.py — ingest_song(query)
  Calls Groq to extract:
    key, time_signature, tempo_feel, difficulty,
    chords[] (symbol + function),
    song_sections[] (name + chord list),
    techniques[], feel
  Returns: SongObject
        ↓
agents.py — run_council(song_object)
  Spawns 4 parallel asyncio tasks, each calling Groq:
    - Theory Agent    → 200-250 word theory section
    - Technique Agent → 200-250 word technique guide
    - Ear Agent       → 200-250 word ear training section
    - Planner Agent   → structured practice plan
  Returns: AgentOutput (4 text fields)
        ↓
chairman.py — synthesise(song_object, agent_output)
  Calls Groq to write chairman_summary + quiz questions
  Resolves chord symbols to chord_keys via chord_definations.json
  Builds PracticeChord[] and SongSection[]
  Returns: LessonDocument
        ↓
router.py — _embed_fingerings_and_functions(lesson, song_chords)
  Looks up each chord_key in fingerings.json
  Embeds FingeringPosition[] on each PracticeChord
  Embeds chord_function (Roman numeral role) on each PracticeChord
  Returns: enriched LessonDocument
```

**LessonDocument schema** (what the frontend receives):
```
lesson_id             string  (UUID, used for tip/revise/quiz calls)
song_title            string
artist                string
key                   string
time_signature        string
tempo_feel            string
overall_difficulty    "beginner" | "intermediate" | "advanced"
chairman_summary      string
theory_section        string
technique_section     string
ear_training_section  string
practice_plan         string
practice_chords[]
  symbol              string  (e.g. "Am7")
  chord_key           string  (e.g. "A_minor7")
  difficulty          string
  available_in_app    bool
  positions[]         FingeringPosition[]
  chord_function      string  (e.g. "ii of V")
song_sections[]
  name                string  (e.g. "Verse")
  chords[]            string[] (symbols)
chord_functions{}     Record<symbol, Roman numeral string>
quiz_questions[]      QuizQuestion[]
```

### Personalization Engine (`personalization/planner.py`)
Builds a daily plan from the user's history:
1. Query `chord_attempts` for recent scores per chord
2. Pick **warmup** chords: highest-scoring, not practiced today
3. Pick **focus** chord: lowest recent average score (most needs work)
4. Pick **new challenge**: chord at the next difficulty level not yet attempted
5. Pick **transition** pair: highest miss-rate pair from `transition_drills`
6. Cap items by skill level (beginner: 5, intermediate/advanced: 7)

### Auth Layer (`auth/`)
- JWT access tokens (15-minute expiry) + refresh tokens (30-day expiry, stored hashed in DB)
- Roles: `student` | `teacher` | `admin` — enforced by `require_role()` dependency
- `build_user_context()` enriches requests with skill level for personalised AI responses
- Passwords hashed with bcrypt via passlib

### Database (`db/`)
- **Engine:** SQLAlchemy 2.0 async with `asyncpg` (Postgres) or `aiosqlite` (local dev SQLite fallback)
- **Migrations:** Alembic — `alembic upgrade head` runs on deploy
- **Models:** `User`, `RefreshToken`, `ChordAttempt`, `TransitionDrill`, `QuizResult`, `TeacherAssignment`
- `DATABASE_URL` env var switches between Postgres and SQLite automatically

## Data Layer

### `data/lessons/` (84 chord folders)
Each folder is named by `chord_key` (e.g. `A_minor`, `G_major`) and contains:
- `metadata.json` — `{ chord, notes[], difficulty }`
- `reference.wav` — reference audio (served at `GET /audio/{chord}`)
- `lesson.mp4` — optional lesson video (served at `GET /video/{chord}`)

### `data/chords/fingerings.json`
Array of fingering objects, one per chord. Each object:
```json
{
  "chord": "A_minor",
  "display_name": "Am",
  "root": "A",
  "quality": "minor",
  "notes": ["A", "C", "E"],
  "positions": [
    { "string": 2, "fret": 1, "finger": 1, "note": "C" },
    { "string": 3, "fret": 2, "finger": 2, "note": "E" },
    { "string": 4, "fret": 2, "finger": 3, "note": "A" }
  ]
}
```

### `data/chords/chord_definations.json`
Chord catalogue used by the Chairman to resolve chord symbols to `chord_key` values.

## Communication Patterns

### Auth / Token Refresh Flow
```
Frontend → attach Bearer token from localStorage
        ↓
Backend  → validate JWT → extract user_id → load user from DB
        ↓
If 401   → frontend calls POST /api/auth/refresh
         → if refresh succeeds → retry original request
         → if refresh fails    → redirect to /login
```

### Chord Practice Flow (home page)
```
1. User picks chord       → POST /learn_chord
2. User records audio     → POST /submit_audio (multipart)
3. Backend evaluates      → { score, evaluation, analysis, fingering_tips }
4. Frontend renders       → FretboardVisualizer + FingeringTipCards + score ring
5. If in song context     → POST /api/council/practice/tip (personalised LLM tip)
```

### Song Lesson Flow
```
1. User searches song     → POST /api/council/generate
2. Frontend renders       → IntegratedLessonView (scrollable single surface)
3. User clicks section    → SongTimeline highlights → ChordMasteryBoard dims non-section chords
4. User clicks chord card → ChordPracticeDrawer slides in
5. User submits audio     → POST /submit_audio + POST /api/council/practice/tip
6. All chords attempted   → POST /api/council/practice/revise (optional lesson revision)
7. Quiz                   → POST /api/council/quiz → MCQ rendered inline
8. Quiz passed            → PlayAlongMode unlocked
```

## Key Architectural Decisions

| Decision | Rationale |
|---|---|
| 4 parallel LLM agents | Shorter, focused prompts produce higher quality than one big prompt; parallelism keeps total latency under ~5s |
| Fingerings embedded at generation time | Frontend never needs a separate fingering request during the lesson; fewer round-trips |
| SQLite fallback for local dev | No Docker dependency; same ORM code works on both SQLite and Postgres |
| Session state in-memory (not DB) | Chord practice sessions are ephemeral; persisting them to DB adds latency with no benefit |
| JWT refresh token rotation | 30-day sessions without re-login; refresh tokens stored hashed so a DB breach can't replay them |
| SWR on frontend | Automatic background revalidation keeps progress stats fresh without manual refresh |
| `X-Session-Id` header | Stateless backend can route requests to the correct `TutorAgent` without requiring cookies |
| Council generates everything in one call | User gets a complete, coherent lesson in one request rather than multiple separate fetches |
