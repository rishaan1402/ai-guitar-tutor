# AI Guitar Tutor Architecture

This document describes the architecture of the AI Guitar Tutor system.
The purpose of this file is to help AI assistants and developers understand how the system is structured.

The system is designed using a **modular service architecture** where each module has a clear responsibility.

---

# High-Level Architecture

The platform consists of five main layers.

Frontend Web Application
↓
Tutor Agent (Session Controller)
↓
Lesson Service
↓
Audio Evaluation Engine
↓
Feedback Engine
↓
Storage Layer

Each layer performs a specific task and communicates through well-defined interfaces.

---

# Frontend Layer

The frontend provides the user interface.

Responsibilities:

• accept chord learning requests
• play lesson videos
• record microphone audio
• send audio recordings to backend
• display tutor feedback

Technologies used:

React / Next.js
Web Audio API

Frontend must not perform any audio analysis.

All audio processing must happen on the backend.

---

# Tutor Agent Layer

The Tutor Agent is the **orchestrator of the system**.

It manages the learning session and coordinates the other services.

Responsibilities:

• interpret user intent
• retrieve lesson content
• control session state
• request chord evaluation
• generate tutoring responses

The tutor agent should behave like a **state machine**.

Session states:

IDLE
TEACHING
WAITING_FOR_PLAY
ANALYZING
FEEDBACK
COMPLETED

The tutor agent should not contain audio signal processing logic.

---

# Lesson Service

The lesson service provides teaching content for chords.

Lessons are stored as structured assets.

Example lesson directory:

data/lessons/G_major/

lesson.mp4
reference.wav
metadata.json

Example metadata:

{
"chord": "G_major",
"notes": ["G","B","D"],
"difficulty": "beginner"
}

Responsibilities of LessonService:

• load lessons from disk
• return lesson metadata
• provide paths to video and reference audio

LessonService should dynamically detect available lessons.

---

# Audio Evaluation Engine

The Audio Evaluation Engine analyzes recorded audio to determine if the chord was played correctly.

This module performs signal processing tasks such as:

• noise reduction
• frequency analysis
• pitch detection
• note extraction
• chord matching

Processing pipeline:

Audio Input
↓
Noise Reduction
↓
Frame Segmentation
↓
FFT Spectrum Analysis
↓
Pitch Detection
↓
Note Matching
↓
Error Detection

Example output:

{
"score": 0.75,
"detected_notes": ["G","D"],
"missing_notes": ["B"],
"issue": "muted_string"
}

Libraries used:

librosa
numpy
scipy

This module must be optimized for performance.

---

# Feedback Engine

The Feedback Engine converts technical evaluation results into tutor responses.

Example:

Input:

{
"issue": "muted_string",
"string": "B"
}

Output:

"Your B string isn't ringing clearly. Try pressing closer to the fret."

Feedback should always be:

• short
• supportive
• beginner friendly

---

# Storage Layer

The storage layer manages lesson assets and user data.

Lesson storage:

Google Cloud Storage or local file system.

User data:

Firestore.

Practice attempts may be stored for analytics and improvement.

---

# API Layer

The backend exposes REST APIs used by the frontend.

Endpoints:

POST /learn_chord
POST /submit_audio
GET /lesson/{chord}

Responses must always be JSON.

---

# Future Architecture Extensions

The system must be designed to support future expansion.

Possible future features:

• additional chords
• chord progressions
• real-time audio analysis
• additional instruments
• mobile applications

Avoid hardcoding guitar-specific logic where possible.

Use modular design so that additional instruments can reuse the same architecture.
